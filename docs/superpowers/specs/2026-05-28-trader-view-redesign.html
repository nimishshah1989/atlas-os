Trader-View Redesign — Master Spec

Status: design-locked · ready for implementation planning  |  Authored: 2026-05-28  |  Vocabulary source: CONTEXT.md §"Cell state vocabulary" (canonical, with WAIT addition committed 2026-05-28)  |  Visual reference: docs/v6/mockup-trader-view.html

1. Problem statement

Atlas v6 today exposes four independent decision frameworks on the same page with no precedence between them:

  1. Cell state + conviction tier — the IC-weighted pattern-match output (POSITIVE / NEUTRAL / NEGATIVE; tier T1-T5)
  2. Predicted excess + confidence — the forecast (e.g. "+7.4% over 3M, 0.71 confidence")
  3. Weinstein stage — current-cycle state classifier (Stage 1/2/3/4)
  4. 5-gate investability — structural filter (strength × direction × risk × sector × market)

These frameworks regularly contradict each other on the same instrument. A stock can have POSITIVE cell + Stage 4 (Weinstein veto). An ETF can pass all 5 gates while Weinstein says "below 30W MA, avoid new positions." The UI renders all four as if they were graded on the same axis, leaving the user to mentally reconcile contradictions on every card.

Fund-manager feedback (2026-05-28): "I am not able to see exactly what the rules are. Some of the conviction names today did not make sense when I looked at them on TradingView." Translation: methodology is not visible, decisions are not coherent, and the system fails the trust-by-transparency test that Atlas was supposed to win.

2. Solution

Collapse the four frameworks into a single Atlas verdict with a fixed precedence ladder. The four frameworks become evidence pillars that explain the verdict, not competing scores. Trader sees one decision; analyst can click through to the math.

  MARKET REGIME   ──┐  (risk-off → block all new positions)
  SECTOR STATE    ──┤  (Avoid → block, OW → boost)
  WEINSTEIN STAGE ──┤  (Stage 4 → veto, Stage 2 → endorse)
  CONVICTION TIER ──┤  (T1/T2 = act, T3-T5 = watchlist)
  CELL ACTION+IC  ──┘  (POSITIVE/NEUTRAL/NEGATIVE; predicted excess sizes)
                ↓
        ONE ATLAS VERDICT
   BUY · ACCUMULATE · WATCH · HOLD · AVOID · SELL · WAIT

3. Vocabulary (CONTEXT.md aligned, with WAIT added)

The cell math produces one of three states. The UI renders an ownership-aware label, plus a fourth derived label (WAIT) when a veto blocks acting on a positive cell.

NEUTRALWATCHHOLDn/a
NEGATIVEAVOIDSELLn/a
WAIT (added to CONTEXT.md 2026-05-28): a display-layer override applied when cell state is POSITIVE but a veto blocks action. The cell row in atlas_signal_calls remains POSITIVE — WAIT is computed at render time. This preserves IC backtest semantics while preventing the trader from acting on a vetoed signal. When WAIT renders, the reason must be named ("Stage 4 vetoes positive cell" or "Risk gate fail: extension > 40%").

4. Verdict derivation

Computed daily as part of the existing MV refresh pipeline. Output: one verdict column per instrument per day, plus a verdict_reason string when WAIT or AVOID/SELL fires.

def derive_verdict(cell_state, weinstein_stage, gates, user_owns):
    # 1. NEGATIVE cells: ownership decides verb
    if cell_state == 'NEGATIVE':
        return ('SELL', None) if user_owns else ('AVOID', None)

    # 2. NEUTRAL cells: holding pattern
    if cell_state == 'NEUTRAL':
        return ('HOLD', None) if user_owns else ('WATCH', None)

    # 3. POSITIVE cells: check vetos before promoting to BUY/ACCUMULATE
    if cell_state == 'POSITIVE':
        # 3a. Weinstein veto
        if weinstein_stage == 4:
            return ('WAIT', 'Stage 4 vetoes positive cell')
        # 3b. Gate veto (any fail blocks)
        failed_gate = next((g for g in gates if not g.passed), None)
        if failed_gate:
            return ('WAIT', f'{failed_gate.name} gate fail')
        # 3c. Stage 3 ambiguity → caution for new buyers
        if weinstein_stage == 3:
            return ('HOLD', 'Stage 3 topping') if user_owns else ('WATCH', 'Stage 3 topping')
        # 3d. Clear path: render BUY/ACCUMULATE
        return ('ACCUMULATE', None) if user_owns else ('BUY', None)

This logic is the single source of truth. Every page (stocks, ETFs, sectors, funds, homepage) renders from the same column.

5. First-called tracking + since-call return

Schema already exists. CONTEXT.md lines 22-29 lock the signal_call_id lifetime:

  • A row in atlas_signal_calls stays open while the cell state remains the same.
  • When state changes, the existing row gets exit_date set and a new row with a fresh signal_call_id opens.
  • entry_date is therefore the canonical "first called X on" date.

What we surface in the UI per instrument:

  • First called <verdict> on YYYY-MM-DD (= entry_date of the currently-open signal_call_id)
  • Days held (current_date − entry_date in trading days)
  • Since-call return (close_today / close_at_entry − 1)
  • Predicted band at 1M / 3M / 6M / 12M (predicted_excess prorated for elapsed time, ± σ_predicted)
  • Drift status (within band / drift above / drift below)

6. Drift detection

This is the accountability lock. Without it, "predicted +7.4%" is a claim we never verify. With it, every signal_call is tracked daily against its own forecast.

Daily compute (per open signal_call)

realized_excess = close_today / close_at_entry − bench_today / bench_at_entry
elapsed_frac    = days_in_position / tenure_days  # 0.0 to 1.0
predicted_today = predicted_excess × elapsed_frac
σ_today         = σ_predicted × √elapsed_frac    # variance scales with time
Z               = (realized_excess − predicted_today) / σ_today

Drift event log

When |Z| > 2, write a row to atlas_drift_event_log with (signal_call_id, realized, predicted, σ, Z, observed_at). This is the SEBI-grade audit trail. Schema exists per CONTEXT.md lines 115-130.

UI surface

  • Within band (|Z| ≤ 1.5): green check, no chip needed
  • Mild drift (1.5 < |Z| ≤ 2): amber chip "drift +1.8σ above predicted"
  • Significant drift (|Z| > 2): red chip "drift −2.4σ — call is failing"

On call close (exit_date set), final Z gets logged regardless of magnitude. This builds the rolling-IC dataset that feeds the Stage 4a weight optimization loop.

7. Weinstein per cap-tier research

A 30-week MA is the literature standard for US large-caps. Indian small-caps move at a different timescale; their full cycle can complete in 12-15 weeks. Applying 30W to a small-cap measures the wrong thing.

Theoretical priors (to be validated)

TierWhy a different lookback may fitFirst-guess lookback
LargeSlow institutional flows, literature standard30W
MidFaster cycles, more rotation20W
SmallRetail-driven, fast accumulation/distribution10W
MicroHigh noise; framework may not applyresearch / exclude

Research methodology (methodology-lock principle 1)

  1. Pre-register the bar per cap_tier × tenure: IC ≥ 0.04 at the chosen tenure (matches the cell IC floor table in CONTEXT.md line 568-576).
  2. Candidate grid: 5W, 10W, 20W, 30W, 50W moving averages × Large / Mid / Small / Micro.
  3. Compute stage classification for each candidate (price-vs-MA + slope sign over N weeks).
  4. Compute forward IC at 1m, 3m, 6m, 12m horizons for each (cap_tier × lookback × stage) cell.
  5. Compute whipsaw rate (proportion of stocks flipping stage within 5 trading days of classification). Lower is better.
  6. Walk-forward validation across at least 5 years of Indian equity data. Out-of-sample IC must remain above floor.
  7. Lock the winners per cap_tier into atlas_thresholds. Update methodology page with the chosen values + the rationale.

Fund-manager point: "capture uptrend faster"

The fund manager flagged that Stage 1→2 transitions are too slow to capture in the current rules. Candidate fix: add a fast-confirm overlay. Promote Stage 1 to Stage 2 when price > chosen_MA AND chosen_MA slope > 0 for 3 consecutive weeks (instead of waiting for full slope inversion). Trade-off: some false-positive Stage 2 calls in exchange for earlier entry. The research workstream should test fast-confirm vs strict-confirm as a parameter.

Visual reference: see the rendered mockup at docs/v6/mockup-trader-view.html. Three cases (clean BUY, conflict WAIT, ownership-aware SELL) shown side-by-side with current layouts.

Layout from top of stock-detail page:

  1. Stock header — symbol, name, sector, cap-tier, current price (1 line)
  2. Verdict block — large verdict pill (BUY / WAIT / SELL etc.), expected return + tier badge, first-called date + since-call return
  3. Why-strip — 4 chips (Cell · Weinstein · Investable · Sector) colored pass/warn/fail; conflict cases add a 5th "Reconcile" chip naming the override
  4. Tracking grid — at-call price, 1M target, 3M target, realized today, "within band" badge
  5. Show the math (collapsed by default) — IC values, gate breakdown, composite components, σ band, cell history
  6. Existing charts and panels below (RS chart, holdings, peer comparison, etc.)

10. Workstreams

A — Weinstein per cap-tier research

Backtest the (cap_tier × MA lookback × stage) matrix to find the optimal MA per cap-tier. Output: locked thresholds in atlas_thresholds, walk-forward validation report, fast-confirm overlay decision.

Dependencies: none. Pre-req: office-hours review of research methodology before kickoff. Effort: 3-4 days.

B — Drift detection wiring

Daily realized-vs-predicted compute, Z-score, drift event log writes, UI chip on every stock page.

Dependencies: none (parallel with A and C). Effort: 2 days.

C — Verdict composition + first-called surfacing

Add combined_verdict + verdict_reason + first_called_at + since_call_return as MV columns. Apply verdict derivation logic from §4.

Dependencies: A must land before C if Weinstein veto rules depend on per-tier thresholds. C can ship with default 30W and migrate to per-tier values once A lands. Effort: 2-3 days.

D — Per-page UI conversion

Apply the trader-view pattern to /stocks/[symbol] first. Hold a checkpoint, then propagate to /etfs/[ticker], /sectors/[name], /funds/[mstar_id], homepage.

Dependencies: C must land first (UI cannot render columns that do not exist). Effort: 5-7 days, page-by-page with checkpoint after /stocks.

E — Methodology page rewrite

Plain-English explainer of: cell math, Weinstein rules (with the rules from A), drift detection, verdict derivation. Fund-manager-grade clarity per the explainer-flywheel ethos.

Dependencies: A must lock first (cannot explain rules not yet decided). Effort: 1-2 days after A.

11. Open questions to lock before plans dispatch

  1. Does Stage 3 veto a positive cell, or only downgrade to WATCH/HOLD? §4 derivation currently treats Stage 3 as a downgrade, not a veto. The alternative is veto = WAIT.
  2. What is σ_predicted calibration? Per-cell from backtest (preferred) or universal heuristic?
  3. Fast-confirm overlay for Stage 1→2: ship by default once research completes, or A/B test against strict-confirm?
  4. Ownership signal source: atlas_paper_portfolio (paper trading) or atlas_user_lots (real positions)? Determines which gets queried in the verdict derivation.
  5. Micro-cap inclusion: exclude from Weinstein framework entirely, or apply research and see if any lookback satisfies the IC floor?
