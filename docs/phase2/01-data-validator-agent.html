1. Why this agent exists

Atlas's frontend renders hundreds of data points per page across screener, deep-dive, sector, fund, ETF, and regime views. Each of those points has a backend lineage: a SQL query, a computed metric, a state classification, a threshold check. The lineages can silently break in dozens of ways:

  • A migration adds a column but the frontend query never updates — silent NULL displays
  • A threshold value changes in the database; downstream caches don't refresh — stale states
  • A compute function divides by zero on holiday runs — inf values render as "Infinity" in the UI
  • Two pages display the same metric using different SQL paths — they disagree by a percent
  • An ETL job partially completes — some rows are today, some are yesterday, the UI doesn't know
  • A type coercion silently truncates a Decimal — a portfolio value shows ₹0 instead of ₹12,34,567

The health audit closed the visible bugs. This agent prevents the next class of bugs — the ones we'd only discover from a user screenshot at 11 PM. It runs continuously, always-on, never tires.

The contract. Every data point shown to a user has a verifiable lineage. The agent traces the lineage, recomputes the value where feasible, compares displayed value to authoritative value, and flags every divergence with severity, evidence, and remediation suggestion.

3. The six classes of integrity issue

Every finding maps to exactly one of these six classes. The taxonomy is closed — if a finding doesn't fit, the taxonomy needs to expand (logged as a discovery).

CLASS 1 Data gaps

Definition: A field expected to have a value is missing — NULL where NOT NULL was expected, missing date in time series, missing instrument in universe, missing column entirely.

Detection method: Schema comparison + completeness queries. The agent maintains an expected-coverage map (e.g., NIFTY 500 stocks should have 252+ rows in atlas_stock_metrics_daily; sectors should be 100% covered each business day).

Examples:

  • Stock RELIANCE has no row in atlas_stock_states_daily for 2026-05-11 but has rows for 2026-05-10 and 2026-05-12
  • Sector "Healthcare" missing from atlas_sector_metrics_daily while other 13 sectors are present
  • rs_velocity column NULL for 30+ instruments on most recent date

CLASS 2 Inconsistencies (frontend ↔ backend divergence)

Definition: A value displayed on the frontend differs from what the backend SQL/API would return if queried fresh. Caching, stale builds, query mismatches.

Detection method: Playwright crawls frontend routes, extracts displayed values; agent queries backend via SQL/API for the same identifiers; compares.

Examples:

  • Screener shows TCS market cap ₹13,45,000 Cr; backend query returns ₹13,47,200 Cr
  • Deep-dive shows Reliance momentum_state="Strong"; backend returns "Improving"
  • Sector page shows IT sector RS=0.82; sector page on different route shows 0.79

CLASS 3 Calculation errors

Definition: A computed value disagrees with the methodology formula when recomputed on the same inputs. Code bug, off-by-one window, wrong divisor.

Detection method: Agent re-implements (or invokes the reference implementation of) the methodology formula on the underlying primitives; compares to stored value.

Examples:

  • EMA-50 on stock X reports 487.20 but recomputing from raw OHLCV gives 489.15
  • Drawdown ratio computed against wrong benchmark
  • RS percentile based on a 252-day window but methodology says 63-day for this timeframe

CLASS 4 Accuracy errors

Definition: The value is mathematically correct given inputs but the inputs are wrong (wrong source, wrong as-of date, wrong corporate-action adjustment).

Detection method: Cross-source verification (compare to yfinance or NSE direct query) on a sample. Audit corporate-action proximity (split/dividend within window).

Examples:

  • Stock close price diverges from NSE official close by more than 0.5% (likely missing split adjustment)
  • NAV used for fund metrics is from T-3 instead of T-1 (stale source)
  • Sector classification uses NSE old taxonomy on stocks reclassified by exchange

CLASS 5 Insensible values

Definition: A value falls outside the universe of physically possible or business-rule-permitted values. Negative volumes, infinite ratios, dates in the future, percentages over 100%.

Detection method: Domain validators per column. Encoded as boundary conditions: volume_state IN ('Quiet', 'Normal', 'Active', 'Frenzy'); 0 ≤ rs_percentile ≤ 1; date ≤ TODAY.

Examples:

  • EMA value of inf from division by zero on holiday vol
  • NAV with date "2027-08-12" (future) in atlas_fund_lens_daily
  • Drawdown reported as 1.45 (145%) — impossible by definition (max -100%)
  • Negative number of holdings in a fund's portfolio

CLASS 6 Incomplete data

Definition: A value is present but lacks the dependencies that would make it meaningful. A computed metric that depends on 252 days of history but only has 80 days, displayed without a caveat.

Detection method: Trace dependency chain. Every primitive declares its minimum-input requirement; agent verifies the requirement is met for every displayed value.

Examples:

  • RS percentile computed for a stock with only 45 days of trading history; UI displays a value but it's not statistically meaningful
  • Fund alignment metrics shown for a fund with no portfolio disclosure in past 90 days
  • Beta-to-benchmark shown for an ETF whose linked index history starts 30 days ago

4. Why Hermes Agent

The validator agent needs: scheduling, tool registry, retries, memory of prior findings, ability to delegate to specialist sub-agents (one per data class), JSON-Schema tool calls, and the option to run on a local model for DPDP-sensitive SQL inspection.

NousResearch's hermes-agent (144K stars, Python, MIT-licensed) provides every one of those primitives. We don't build the agent loop; we register tools and ship.

Capability neededHermes provides it via
Tool registry with JSON Schema validation<tools> block in system prompt; built-in validator
Scheduled execution (nightly)Cron tool + agent scheduling primitive
Retry on failureBuilt-in retry handler with exponential backoff
Memory across runsHermes context engine + persistent memory layer
Subagent delegation (one agent per data class)delegate tool — spawn focused subagents
Model swap (Claude ↔ local Llama)hermes model CLI flag; model-agnostic
Trajectory loggingShareGPT-format JSONL output; audit-friendly
Alternative considered: TauricResearch's TradingAgents (74K stars) uses LangGraph for multi-agent finance workflows. Stronger out-of-box financial reasoning but US-equity-focused. Choosing Hermes for the validator because the validator is a general-purpose tool-using agent, not a finance reasoning agent. We retain TradingAgents as the framework to fork later for SP07 specialist agents.

5.1 The orchestrator

One Hermes agent runs as the orchestrator. It receives an invocation (cron-triggered or on-demand), pulls the current route map from memory, decides which subagents to delegate to, waits for their findings, and routes to the reporter. The orchestrator is small — its job is decomposition, not validation.

5.2 The five subagents

  • Route Crawler — Playwright-driven. Loads every frontend route, captures DOM, extracts visible data points with their data-attribute lineage tags (see §10.2 for lineage tagging convention).
  • Schema Validator — Pure SQL. Compares expected coverage map to actual row counts per (table, date) pair. Flags missing dates, missing instruments, NULL clusters.
  • Cross-page Consistency — Reads metric values from multiple frontend routes; ensures the same logical metric (e.g., "TCS market cap") returns same value across routes.
  • Calc Verifier — Re-runs the methodology formula for sampled rows. Compares to stored value. Flags off-by-one window errors, wrong divisor, wrong adjustment.
  • Sensibility Validator — Runs domain constraint checks on every numeric column. Flags inf, NaN, out-of-range, future-dated.

5.3 The reporter

Receives findings from all subagents. Deduplicates (same finding from two subagents = one report row). Classifies severity (see §8). Writes to atlas_validator_findings table. Posts P0/P1 to Slack. Updates the admin dashboard.

7. Validation rules in detail

7.1 Lineage tagging convention (load-bearing)

For the validator to trace frontend values back to backend sources, every displayed number must carry a lineage tag. Convention (added incrementally to existing components):

<span data-lineage="atlas.atlas_stock_metrics_daily.close|date={date}|instrument_id={id}">
  ₹3,402.50
</span>

The lineage tag is structured: schema.table.column|filter_key=value|.... The validator parses it, constructs the canonical SQL query, executes it, and compares the result to the displayed value.

Backfilling lineage tags is a multi-week task. The validator v1 starts with a small set of high-value pages (screener, stock deep-dive, sector page, regime banner) and grows coverage as components are touched in regular development. New components added to Phase 2 sub-projects must include lineage tags from day one — this is enforced by a pre-merge check.

7.2 Coverage expectations (the expected-coverage map)

The validator maintains a YAML expected-coverage map. Format:

tables:
  atlas_stock_metrics_daily:
    expected_instruments: nifty_500_membership@as_of_date
    expected_dates: business_days(start=2016-04-07, end=T-1)
    null_allowed_columns: [bsa_volume, derivatives_oi]
    null_forbidden_columns: [close_approx, ema_10, rs_3m]
    tolerance: 99.5%  # i.e., 99.5% of expected rows must exist
  atlas_sector_metrics_daily:
    expected_sectors: [Auto, Banking, Capital Goods, ...]
    expected_dates: business_days(...)
    null_forbidden_columns: [bottomup_rs_3m, sector_state]
    tolerance: 100%
  atlas_fund_lens_daily:
    expected_instruments: regulated_fund_universe@as_of_date
    expected_dates: nav_business_days(...)
    null_allowed_columns: [aligned_aum_pct, neutral_aum_pct, avoid_aum_pct]  # disclosure-gated
    tolerance: 90%  # not all funds disclose

7.3 Sensibility rules (per-column constraints)

Column patternConstraint
*_stateString value must be in the per-state enum (e.g., rs_state ∈ {Leader, Strong, Emerging, Improving, Weakening, Lagging, Weak, Avoid})
*_pct (percentages stored as fractions)-1 ≤ value ≤ 5 (allow -100% return floor, 500% max single-period return — beyond this is suspicious)
*_pct (display percentages, 0-100 scale)0 ≤ value ≤ 100
*_percentile0 ≤ value ≤ 1
datedate ≤ TODAY and date ≥ HISTORICAL_START_DATE
volumevalue ≥ 0 and value < 1e12 (1 trillion shares = suspicious)
aum_*value ≥ 0
Any numericvalue is finite (no inf, no NaN)
ema_*value > 0 for price-derived EMAs

7.4 Cross-page consistency map

Each metric that appears on multiple pages is registered as a consistency check. The validator extracts the metric from each page (using lineage tags) and asserts they are equal within tolerance.

consistency_checks:
  - metric: "TCS market cap"
    pages: [/stocks, /stocks/TCS, /sectors/IT (constituents view)]
    tolerance: 0.5%
  - metric: "Current regime state"
    pages: [/, /regime, /stocks (banner), /funds (banner)]
    tolerance: exact_match
  - metric: "Nifty 500 close"
    pages: [/regime, /sectors, /]
    tolerance: 0.1%

7.5 Calculation re-verification

For a stratified random sample (10 stocks × 5 dates = 50 cells per run), the Calc Verifier subagent independently recomputes each metric from raw inputs and compares to the stored value. Sample composition: 5 large-cap, 5 small-cap; 5 recent dates, 5 older dates. Tolerance: 0.01% absolute for prices, 0.5% relative for derived ratios.

8. Reporting format and severity

8.1 Severity matrix

P0 — Block

Insensible value visible to user (inf, NaN, future date). Cross-source price disagreement > 5%. Affects > 10% of users.

P1 — Fix today

Frontend ↔ backend divergence > tolerance. Cross-page inconsistency on a key metric. Calculation error on widely-displayed value. Affects 1-10% of surface.

P2 — Fix this batch

Data gap below tolerance. Missing lineage tag. Stale cache. Affects <1% but should be cleaned.

P3 — Track

Suggestion / improvement. Lineage tag could be more specific. Coverage map could expand. Not user-facing.

8.2 Finding row schema

atlas_validator_findings:
  id              UUID PRIMARY KEY
  run_id          UUID (FK to atlas_validator_runs)
  finding_class   VARCHAR(32) -- one of the 6 taxonomy classes
  severity        VARCHAR(8)  -- P0 | P1 | P2 | P3
  route           TEXT        -- frontend URL where finding surfaced (NULL if backend-only)
  surface         TEXT        -- table/column or component identifier
  identifier      TEXT        -- e.g., "instrument_id=RELIANCE,date=2026-05-11"
  expected_value  TEXT        -- canonical/correct value
  actual_value    TEXT        -- what was displayed/stored
  delta_abs       NUMERIC     -- nullable (only for numeric findings)
  delta_pct       NUMERIC     -- nullable
  evidence        JSONB       -- agent trajectory + tool-call chain
  remediation     TEXT        -- agent's suggested fix
  first_seen      TIMESTAMPTZ
  last_seen       TIMESTAMPTZ
  resolved_at     TIMESTAMPTZ NULL
  resolved_by     TEXT NULL
  notes           TEXT NULL

8.3 Output channels

  • Postgres: All findings written to atlas_validator_findings
  • Slack: P0 + P1 posted to #atlas-data-alerts with one-line summary + link to dashboard
  • Admin dashboard: /admin/validator renders all findings, filterable by severity / class / surface / date
  • CLI: python -m atlas.agents.validator --report dumps latest findings as markdown for inclusion in standup notes
  • Pre-deploy gate: scripts/preflight_validator.py runs validator and exits non-zero if P0/P1 findings exist — wireable into deploy scripts

9. Deployment and scheduling

9.1 Where it runs

Hermes agent process runs on the EC2 host alongside the Atlas backend. Reasons: same VPC as Supabase (low-latency SQL), same machine as Atlas API (low-latency localhost API calls), Playwright headless browser fits comfortably in t3.large's 8GB RAM during scheduled windows.

Process model: a systemd service (atlas-validator-agent.service) running a thin Python wrapper that starts Hermes in scheduled mode. Idle most of the day; active during the four scheduled windows (see §9.2) and on demand.

9.3 Pre-milestone hook

Per 00-master-plan.html §7.1, every Phase 2 sub-project kickoff includes a validator run as a pre-flight check. The hook:

# before kickoff
python -m atlas.agents.validator --quick --fail-on=P0,P1
# Exit 0 = baseline clean, safe to proceed
# Exit non-zero = fix P0/P1 first, then re-run

  • Tune tolerances based on false-positive rate (target: < 5% FP)
  • Document common-finding remediation patterns
  • Expand lineage tag coverage to 80% of frontend surface
  • Update 00-master-plan.html §7 to mark validator as operational
Sequencing. Phases A through E are dependency-ordered (each builds on the previous). Phase F requires real production-data exposure across multiple nights to tune for false positives — that part runs on calendar time regardless of how fast the code lands. The validator can be built in parallel with Phase 2 Wave 1 sub-projects (SP01, SP02, SP03) since they own different files.

11. Operating procedures

11.1 Triaging findings

  1. P0: Pager-level. Drop current work, investigate immediately. P0 findings should never persist > 1 hour.
  2. P1: Same-day fix. Investigate root cause; if it's a data issue re-run pipeline; if code, fix and deploy.
  3. P2: Add to the next batch. Group similar findings into one fix.
  4. P3: Track in dashboard. Bulk-clean monthly.

11.2 Resolving a finding

Mark resolved_at + resolved_by in atlas_validator_findings via the admin UI. Optionally add notes explaining the root cause. The next agent run sees the finding is resolved; if it re-detects the same issue, the finding's last_seen updates without creating a duplicate row (deduplicated by finding_class + surface + identifier).

11.3 Tuning tolerances

If a finding is consistently flagged but not actually a problem (e.g., a corporate-action edge case), update the tolerance in the relevant YAML config. The PR doing so must include 5+ examples of why the tolerance is being relaxed — never quietly silence findings.

11.4 Adding a new validation rule

Three steps:

  1. Add the rule to the appropriate YAML (sensibility, coverage, consistency) or subagent code
  2. Add a test case in tests/agents/validator/ with a known-bad fixture proving the rule fires
  3. Run the validator once in --dry-run mode to confirm the rule doesn't fire on production data unexpectedly

11.5 What happens when the validator itself is wrong

Findings are agent output — not infallible. If a P1 alert turns out to be a false positive, the investigator updates the rule and adds a regression test. The agent's findings are inputs to human judgment, not autonomous actions. No finding ever auto-modifies Atlas data.
