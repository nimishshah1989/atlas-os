1. Why Phase 2 exists

Atlas today is correct, well-tested, and architecturally clean. The 2026-05 health audit closed all P0/P1 issues. But correctness alone is not the product. Today Atlas is a static interface that:

  • Ingests OHLCV, NAV, fundamental data from JIP and AMC sources
  • Computes deterministic states (RS, Momentum, Risk, Volume) using rule-based thresholds
  • Renders results in dashboards, screeners, and deep-dive pages

What Atlas does not yet do, and what Phase 2 builds:

  1. Know which signals are predictive — every signal is treated as equal weight. Phase 2 measures information coefficient on every signal and demotes weak ones.
  2. Explain itself — there is no narrative layer. Phase 2 adds Claude-generated daily briefs that read 50+ metrics and produce SEBI-safe analyst-grade prose.
  3. Reason interactively — there is no chat surface, no agent that can answer "show me sectors rotating from defensive to cyclical given current regime." Phase 2 adds a Hermes-orchestrated agent runtime over Atlas's existing endpoints.
  4. Improve itself — thresholds are tuned manually. Phase 2 runs nightly parameter-space tournaments and proposes improvements that beat current rules on rolling IC.
  5. Be measurably accurate — frontend and backend can silently diverge. Phase 2 deploys a continuous data-integrity agent that catches gaps, calculation errors, and inconsistencies before users see them.

The unifying principle: every layer that produces a number must also produce a confidence. Every signal carries an IC. Every state transition has a measured base rate. Every Claude-generated narrative includes an audit trail. The system is honest about what it knows and what it doesn't.

2. The four pillars

The nine sub-projects map onto four engineering pillars. Pillar names appear as labels throughout this document.

Pillar I — Validation

Every signal is measured for predictive power on real forward returns. Weak signals are demoted, strong signals are weighted higher in composites. No assumption is permitted to stay unmeasured.

Pillar II — Intelligence

Above the deterministic state layer, Atlas produces graded scores, regime-conditioned composites, multi-timeframe confluence, and Claude-authored narratives. Inputs to the agent layer.

Pillar III — Agency

Atlas reasons over its own data. Hermes-orchestrated agents compose Atlas endpoints into research workflows. OpenBB workspace integration makes Atlas a first-class analyst tool. Data-integrity agent runs continuously.

Pillar IV — Learning

The system tunes itself. Nightly simulation tournaments across threshold and weight parameter spaces. Winners get promoted with full audit trail. Atlas evolves with the market, not against it.

Signal Validation Lab

Pillar I · Information Coefficient measurement on the full decision_state composite

✓ Shipped 2026-05-12IC gate FAIL on v1 composite — informative, drives SP04 redesignUnblocks: SP04, SP06
Objective
Measure whether Atlas's full decision_state composite (RS state + Momentum state + Risk state + Sector state + Regime state) is predictive of 5-day, 21-day, 63-day forward returns. Establish baseline IC. Identify which sub-signals contribute predictive power and which add noise.

Every downstream "intelligence" feature depends on signals having genuine predictive content. Building Claude briefs, agents, and continuous simulation on top of unmeasured signals is automating confident wrong answers. This is table-stakes validation.

Deliverables
  1. Persistent atlas_signal_ic table: signal_name, timeframe, period (5/21/63), rolling window (3M/6M/12M), mean IC, IC t-stat, IC decay, turnover, computed_at
  2. CLI: python scripts/run_signal_validation.py --signals decision_state --periods 5,21,63 --rolling-window 6M
  3. Markdown tearsheet output under output/validation/decision_state_2026-05-11.md with quantile returns, IC distribution, decay curves

  • Risk: IC is below threshold for current rules. Mitigation: That's an answer, not a failure. Drives sub-project 03 (composite re-design).

Validation strategy
Three independent checks: (1) compute IC on a single known-strong signal (12m return reversal) and verify Alphalens reports IC > 0.06; (2) compute IC on randomized signal labels and verify IC ≈ 0 (sanity); (3) cross-validate against vectorbt portfolio returns from Q5-only positions.

Pillar II · Multi-timeframe RS confluence, IC-weighted composites, regime conditioning

✓ Stage 3 Shipped 2026-05-12T1 + T3 industry-grade (IC≥0.05)T2/T4/T5 baseline — Stage 4 auto-tuning nextLive at /stocks, /intelligence, /stocks/[symbol]

Tier 1 (NIFTY 50) holdout IC 0.0511 (t=6.19), Tier 3 (151-300) holdout IC 0.0538 (t=8.53) — both reach industry-grade. Objective

Agents (SP07), briefs (SP05), and simulation (SP06) all consume the graded score, not the binary state.
Tech stack
Pure Python + Pandas. Reads IC weights from atlas_signal_ic (SP01). Writes new graded scores to atlas_stock_intelligence_daily. Computed during nightly pipeline; cached in materialized views.

  rs_confluence_score    (0..1, IC-weighted)
  momentum_confluence    (0..1)
  conviction_score       (0..1, composite of above)

Daily Atlas Brief

Pillar II · Llama 3.3 70B-authored daily market narrative with SEBI-safe language

✓ Shipped 2026-05-12Live at /intelligence/daily-briefPivoted: Groq Llama 70B not Claude Sonnet (free tier; SP04 dep dropped)
Objective
Build a structured-input narrative-output pipeline that reads ~50 metrics from Atlas's materialized views, packages them into a canonical DailyMarketContext object, and asks Claude Sonnet 4.6 to produce a 200–280 word morning market brief in plain prose. Why structured-input not RAG
RAG is for retrieval over corpora. Atlas has 50 metrics — that's a structured object, not a retrieval problem. Bundling all metrics into one prompt context (~1200 tokens) gives Claude full picture in one call. RAG adds latency, complexity, and retrieval errors for no benefit.

Continuous Simulation Engine

Pillar IV · Nightly parameter-space tournaments — threshold tuning loop

Wave 3Depends: SP01 + SP04Heaviest compute
Objective
Run nightly tournaments where every tunable threshold (RS percentile cutoffs, sector overweight thresholds, regime VIX bounds) competes against ±10%, ±20%, ±30% variants on rolling 6-month IC. Winners get proposed as candidate updates with full audit trail. The user reviews and promotes; nothing auto-deploys. This is the "self-improving" loop you described.
Why this defines Phase 2
A static rule book is what Atlas is today. A rule book that updates itself based on measured performance is what "alive, evolving, learning" means. The simulation engine is the engine of that evolution — quantitative, audited, never auto-deployed, always proposed.
Tech stack
vectorbt (already in simulation extras). Optuna for parameter-space exploration (existing optimizer extras). Reads from atlas_thresholds + atlas_stock_states_daily. Writes to atlas_simulation_results + atlas_threshold_proposals.
Files to create
  • atlas/intelligence/simulation/__init__.py
  • atlas/intelligence/simulation/tournament.py — runs N variants per threshold per night
  • atlas/intelligence/simulation/backtest_runner.py — vectorbt wrapper
  • atlas/intelligence/simulation/scorer.py — IC + Sharpe + turnover scoring
  • atlas/intelligence/simulation/proposer.py — winner detection + audit row
  • migrations/versions/038_simulation_tables.py
  • scripts/m9_nightly_tournament.py
  • frontend/src/app/admin/proposals/page.tsx — review UI
Tournament logic
  1. For each threshold in atlas_thresholds: generate 5 variants (current, ±10%, ±20%, ±30%)
  2. Re-run state classification on the last 6 months with each variant
  3. Compute IC on resulting decision_state against forward returns
  4. Compute portfolio metrics: Sharpe, max DD, turnover
  5. Score variant: score = 0.6×IC + 0.3×Sharpe − 0.1×turnover_penalty
  6. If variant beats current by > 5% on score AND p-value of IC difference < 0.05: propose
  7. Write proposal to atlas_threshold_proposals with full evidence trail
Output / deliverables
  1. atlas_simulation_results — every variant tested, every night, with full metrics
  2. atlas_threshold_proposals — proposed changes with evidence + Claude-authored rationale
  3. Admin dashboard /admin/proposals — review queue, accept/reject UI, side-by-side variant comparison charts
  4. Accepted proposals update atlas_thresholds via existing audit-tracked UPDATE path (migration 023 trigger handles audit log)
Success criteria
  • Tournament completes in < 90 minutes for full threshold catalog × 5 variants
  • At least one statistically significant proposal per month on rolling 6-month IC
  • Accepted proposals show measurable post-acceptance IC improvement (real, not overfit)
  • Zero auto-deployment — every change is human-approved
Risks & mitigations
  • Risk: Multiple-comparison statistical bias — testing 50 variants × 30 thresholds inflates false positives. Mitigation: Bonferroni correction on p-values; minimum effect size threshold; rolling 6-month validation.
  • Risk: Compute time exceeds nightly window. Mitigation: parallelize via Optuna's distributed mode; only re-run thresholds touched in last 30 days.

Objective

6. Key decisions and tradeoffs

6.1 Why we validate before we intelligence

The natural instinct is to build the wow-layer first — Claude briefs, agents, intraday dashboards. The research came back unambiguous: signals without measured IC are unverified opinions. Building intelligence atop unmeasured signals is automating confident wrong answers at scale. SP01 gates everything else.

6.2 Why pg_cron materialized views, not Redis

At Atlas's scale (750 stocks, 100 ETFs, 20 sectors), pg_cron + materialized views deliver sub-3ms reads with zero new infrastructure, zero dual-truth risk, full SQL JOIN capability. Redis becomes valuable only with intraday tick data — and even then, single-source-of-truth (Postgres) with a write-through cache is preferred over independent state. We add Redis in SP08 if and only if KiteConnect intraday data requires it.

6.3 Why Claude Sonnet 4.6, not Opus or Haiku, for daily briefs

Haiku misses subtle market nuance — wealth managers will notice. Opus is 3x cost for marginal gain on this constrained narrative task. Sonnet 4.6 is the right model — better than Haiku on financial reasoning, cheaper than Opus, with prompt caching to reduce repeat-run cost.

6.4 Why Hermes Agent over building our own loop

Hermes is a 144K-star production runtime with: tool registry, JSON Schema validation, scheduling, memory layer, subagent delegation, multi-model support (including local Llama for SEBI sensitivity). Building this from scratch reinvents a complex production-tested loop. Adopting Hermes and registering Atlas tools as plugins keeps us focused on the financial intelligence layer instead of the agent runtime.

6.5 Why we propose threshold changes, never auto-apply

The audit-trail trigger on atlas_thresholds (migration 023) is the load-bearing compliance artifact. Every change must be human-reviewed and human-applied. The simulation engine is allowed to be smart; it is never allowed to be unilateral. Every accepted proposal carries the simulation evidence, the Claude-authored rationale, and the human approver's identity.

6.6 Why we don't switch databases (TimescaleDB / Kafka)

Supabase Postgres is the load-bearing primary store. Migrating to TimescaleDB or adding Kafka adds operational surface for marginal speed gain at current scale. We add capabilities (pg_cron, Redis as cache) without replacing the foundation. If Supabase becomes the bottleneck — measured, not assumed — we revisit.

  • Decimal for money, always — no floats

Halt sub-project work and reassess if:

  • Data Integrity Validator surfaces a P0 finding on production data
  • A sub-project's success criteria are not measurably improving by the end of its time budget
  • A research conflict (like the TimescaleDB/Supabase question) emerges mid-implementation — resolve via evidence, not opinion
