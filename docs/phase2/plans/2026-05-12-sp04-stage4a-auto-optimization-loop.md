# SP04 Stage 4a — Conviction Auto-Optimization Loop Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Required review gates per CLAUDE.md:** This plan MUST pass `/plan-eng-review` AND the admin-UI portion MUST pass `/plan-design-review` BEFORE execution begins. After execution, `/review` runs on the diff and `/qa` validates on production.

**Goal:** Close the loop on the conviction composite. Nightly: compute rolling out-of-sample IC per (tier, signal); generate a candidate weight-set proposal when the current weights drift materially from measured IC; surface in `/admin/composite-proposals` with diff view; on FM approval, write new active weights using 15% Bayesian smoothing toward the candidate.

**Architecture:** Two new audit-tracked tables (`atlas_signal_ic_rolling`, `atlas_weight_proposals`). New module `atlas.intelligence.conviction.optimization` with four pure files: ic_monitor, candidate_generator, smoothing, persistence. Two CLIs (`recompute_signal_ic.py`, `generate_weight_candidates.py`) chained into the nightly. One FastAPI admin route for approve/reject/snooze. One Next.js admin page that reads server-side with the new query layer.

**Tech Stack:** Python 3.12, Pandas, SQLAlchemy 2.0, FastAPI, Supabase Postgres, Next.js 15 + Tailwind.

**Out-of-scope (Stage 4b/4c):**
- Per-regime weight conditioning (5 tiers × 4 regimes = 20 weight sets)
- Auto-revert when live IC drops below 50% of backtest IC for 60 days
- Hit-rate visibility per stock on the deep-dive
- Drift alerts in Slack / daily brief

These are valuable but require the closed-loop infrastructure first.

## Task 2: `ic_monitor.py` — rolling IC measurement

**Files:**
- `atlas/intelligence/conviction/optimization/ic_monitor.py`
- `tests/intelligence/conviction/optimization/test_ic_monitor.py`

Process:
1. Read all instruments in each tier on `as_of_date`.
2. For each (tier, signal, lookback_window=90, forward_horizon=21):
   - Pull historical (signal_value, forward_return) pairs.
   - Compute Spearman IC + t-stat.
3. UPSERT to `atlas_signal_ic_rolling`.

Implementation will reuse the IC primitive in `atlas/intelligence/validation/encoding.py` if compatible; otherwise use `scipy.stats.spearmanr`.

## Task 3: `candidate_generator.py` — propose new weight sets

**Files:**
- `atlas/intelligence/conviction/optimization/candidate_generator.py`
- `tests/intelligence/conviction/optimization/test_candidate_generator.py`

Algorithm (v1, conservative):
1. Load current active weights from `atlas_signal_weights`.
2. Load rolling IC for the past 30 days per (tier, signal) from `atlas_signal_ic_rolling`.
3. For each tier:
   - Compute proposed weight per signal = `|mean rolling IC|` (negative-sign signals automatically flip).
   - Normalize to sum to 1.0.
   - Compare to current weights: if max-element abs delta < 0.05, skip (not material).
4. Build proposal payload with side-by-side weights, predicted holdout IC (sum of weight × |IC|), rationale string.
5. Insert into `atlas_weight_proposals` with `status='pending'`. If a pending proposal already exists for that (tier, regime), mark old one `superseded` and insert new.

Tests cover: weight renormalization sums to 1.0, anti-predictive signals correctly flip, no-change suppression below threshold, supersede logic.

## Task 4: `smoothing.py` — Bayesian blend on approval

**Files:**
- `atlas/intelligence/conviction/optimization/smoothing.py`
- `tests/intelligence/conviction/optimization/test_smoothing.py`

Pure function: `blend_weights(current: dict[str, Decimal], proposed: dict[str, Decimal], lambda_: Decimal = Decimal("0.15")) -> dict[str, Decimal]`

```
blended[signal] = (1 - λ) * current[signal] + λ * proposed[signal]
```

Then re-normalize blended weights to sum to 1.0 (so rounding errors don't drift the total).

Tests: lambda=0 returns current verbatim; lambda=1 returns proposed verbatim; lambda=0.15 produces 0.85/0.15 blend; weights sum to 1.0 within 1e-9.

## Task 5: `persistence.py` — apply proposals

**Files:**
- `atlas/intelligence/conviction/optimization/persistence.py`
- `tests/intelligence/conviction/optimization/test_persistence.py`

Functions:
- `upsert_ic_measurement(engine, row)` — used by Task 2
- `insert_proposal(engine, payload)` — used by Task 3
- `apply_proposal(engine, proposal_id, reviewer, notes)` — atomic transaction:
  1. SELECT proposal; assert status='pending'.
  2. Load current active weights for that tier (regime='all').
  3. Blend with smoothing module.
  4. UPDATE existing weights: SET effective_to=CURRENT_DATE WHERE effective_to IS NULL.
  5. INSERT new weight rows with effective_from=CURRENT_DATE+1, blended values, approved_by='fm-approval'.
  6. UPDATE proposal SET status='approved', applied_weights=blended_dict, applied_at=NOW(), reviewed_by, reviewed_at, review_notes.
  7. The next nightly conviction run picks up the new weights via the existing `load_active_weights` flow.
- `reject_proposal(engine, proposal_id, reviewer, notes)` — UPDATE status='rejected'.
- `snooze_proposal(engine, proposal_id, reviewer, until_date)` — UPDATE status='snoozed' + review_notes including until_date (snoozed proposals are re-surfaced by the generator on a later date).

Tests cover: apply transitions weights table correctly with effective_to bookend; reject does not touch weights; snooze stores until_date.
