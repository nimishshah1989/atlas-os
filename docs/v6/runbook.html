Why this document exists

v6 is a 10-phase build with 8 bounded contexts, 5+ parallel lanes, and an open-ended calendar. Sessions will be days or weeks apart. Without a deliberate context-and-memory regime, the system loses coherence — decisions drift, terminology re-litigates, the same bug gets diagnosed twice. This runbook is the discipline that prevents that.

01The five context layers

Atlas v6's context lives in five layers, each loaded at a different cadence. Knowing what lives where tells you where to add new information and where to look for it.

Layer 1 — Foundational (loaded every session, never optional)
  • ~/.claude/CLAUDE.md — global engineering standards (fintech rules, hook-enforced)
  • CLAUDE.md — project-specific rules (modulith, file-size limits, threshold discipline)
  • CONTEXT.md — the domain glossary (created 2026-05-24 by /grill-with-docs)
  • ~/.claude/projects/-Users-nimishshah-Documents-GitHub-atlas-os/memory/MEMORY.md — auto-memory index

Total budget: ~3-5KB read per session. Always loaded; never skipped.

Layer 2 — Reference docs (loaded on demand, relevant to active phase)
  • ~/.gstack/projects/atlas-os/ceo-plans/2026-05-24-atlas-v6-product-spec.md — CEO plan
  • ~/.gstack/projects/atlas-os/eng-plans/2026-05-24-atlas-v6-eng-review.html — eng review
  • <consolidation>/docs/atlas-signal-discovery/methodology-lock-2026-05-23.md — methodology lock
  • <consolidation>/docs/atlas-signal-discovery/ATLAS_PRODUCT_SPEC_2026-05-24.html — product spec
  • docs/v6/runbook.html — THIS document

Total budget: ~30-50KB per phase. Read the ones relevant to what you're doing.

Layer 3 — Workplan / chunk specs (forge-style)
  • docs/v6/phases/{phase}/ — one folder per phase
  • docs/v6/phases/{phase}/{chunk}.md — per-chunk specs (acceptance criteria, tests, files-touched)
  • docs/v6/TODOS.md — deferred items (T1-T9 from eng review)
  • (optional) orchestrator/plan.yaml + orchestrator/state.db if using /chunkmaster + forge runtime

Created by /to-issues + /chunkmaster. Each chunk should fit one session.

Layer 4 — Per-session checkpoints (continuity primitive)
  • ~/.gstack/projects/atlas-os/checkpoints/{date}-{branch}.md — written by /context-save at session end
  • ~/.gstack/projects/atlas-os/timeline.jsonl — append-only session timeline (skill, branch, outcome)
  • Restored by /context-restore at session start

Each session ends with /context-save; each session starts with /context-restore if the gap is >1 day.

Layer 5 — Audit trail (immutable, append-only)
  • decisions.jsonl — hash-chained log of every substantive edit (Ruflo-managed; hook-enforced)
  • .ruflo/adr/ — Architectural Decision Records for hard-to-reverse calls (rare; only when criteria met)
  • ~/.gstack/projects/atlas-os/{branch}-reviews.jsonl — every /review, /codex, /plan-eng-review entry
  • ~/.gstack/analytics/ — skill usage + telemetry (local)

Never modify after the fact. SEBI RA registration may require these for inspection.

02The four pre-build gates (do these in order)

Before fresh main is cut and Phase 0 begins, four planning gates must close. Each gate produces a load-bearing artifact that downstream phases consume.

Gate 1/plan-design-review on the eng review HTML

Owns UX-readiness for Phase 6. Resolves multi-tenure headline UX, regime-gating chip, SP10 panel visual fit with new scorecard, accessibility, mobile-first vs desktop-first defaults.

Output: design plan at ~/.gstack/projects/atlas-os/design-plans/2026-05-XX-atlas-v6-design-plan.html.

Status: NEXT

Gate 2/to-issues Pass 1 — Phase 0.5 research lanes

Decomposes Lane A1 (0.5a large-cap fix), A2 (0.5b survivorship), A3 (0.5c walk-forward), A4 (0.5d friction + 0.5e Large-cap BUY), A5 (0.5f, 0.5g, 0.5h-prime, 0.5h), B (Phase 1 inventory), C (0.5i SEBI RA), D (Phase 6 wireframes), E (migrations 080-081) into independent tracer-bullet issues.

Output: ~15-20 GitHub issues with triage labels + acceptance criteria.

Status: pending Gate 1

Gate 3/codex:adversarial-review on the consolidated plan

Outside-voice challenge against the CEO plan + eng review + design plan + grill resolutions. Catches what both prior reviews missed.

Status: Codex quota was walled in the CEO review session. RUN WHEN QUOTA RESETS. Fallback: dispatch Claude subagent with fresh context.

Gate 4/to-issues Pass 2 — Phase 1-9 decomposition

Once design plan is locked, decompose Phase 1 (inventory) through Phase 9 (macro overlay) into ~30-50 chunks. Each chunk maps to one issue + one PR + one /ship cycle.

Output: Full issue board; orchestrator/plan.yaml if running forge-runner.

Only after all four gates close does Phase 0 (fork preservation + fresh main) begin. Skipping a gate to "save time" is the failure mode this discipline exists to prevent.

04The per-chunk execution loop

Every implementation chunk follows the same loop. This is the v6 build's heartbeat.

Step 1Mini eng review on the chunk spec

Don't run full /plan-eng-review on every chunk — it's overkill. Instead: read the chunk spec, identify the 3-5 files that'll be touched, identify regression risks, identify which test fixtures matter. ~5 minutes.

Skill: /plan-eng-review only if the chunk is >1 day of work or crosses bounded-context boundaries. Otherwise just think.

Step 2TDD — write tests first against frozen snapshots

Per memory feedback_lift_aggressively_lean_fast.md + global testing rules: tests assert behavior. For v6, every feature math / cell rule / portfolio FSM test uses the small (100-instrument) frozen snapshot. The full snapshot is for nightly only.

Skill: /tdd for new modules; for lifts, write regression tests against v5 behavior first.

Step 3Implement vectorized + Pydantic-typed

Per Principle 2 (eng review §2.2): numpy/pandas vectorized; SQL set ops where data lives in Postgres; Decimal for money; no dict[str, Any] at module boundaries. Branch ≤ 5 per function.

Step 4Run tests + verify regression gates

Locally: pytest on the affected module. CI: look-ahead audit, IC regression, friction golden, LLM eval (if agents/ touched), SEBI guard (if agents/ touched). Per /grill-with-docs Q11 D6 these are non-negotiable CI gates.

Step 5Pre-landing review

/review on the diff (architectural + tests + perf). Then /codex review for independent challenge. Both required for non-trivial PRs.

Step 6Ship

/ship bundles: tests pass, VERSION bump, CHANGELOG entry, commit, push, PR. Then /land-and-deploy for merge + canary verification.

Step 7Update CONTEXT.md if any term semantics changed

If the chunk resolved a previously-ambiguous term or introduced a new one, append to CONTEXT.md in the same commit. Glossary drift is the #1 way multi-session builds lose coherence.

Step 8/context-save at session end

Captures: branch, active chunk, what was done, what's next, decisions made this session, anything uncertain. Read by /context-restore next session.

05Worktree discipline

Eng review §08 + grill mid-Q6 dictate 8 lanes (A1, A2, A3, A4, A5, B, C, D, E) of which 5-6 can run in parallel post-Phase-0. Parallel lanes mean parallel git worktrees.

Naming convention

# Created by /gstack worktree or `git worktree add`
worktree-v6-{lane}-{phase}/
  Examples:
  worktree-v6-A1-largecap-fix/      # 0.5a recovery work
  worktree-v6-A2-survivorship/      # 0.5b
  worktree-v6-E-migrations/         # 080-081 schema
  worktree-v6-D-frontend-wires/     # Phase 6 wireframes

Merge ordering

  1. E (schema) merges first. Migrations 080-081 must land before F (compute backend) can begin.
  2. A1 (large-cap fix) merges before A4 per grill mid-Q6.
  3. A2 (survivorship) merges before A3.
  4. A4 + A5 merge before F.
  5. F merges before G + H.
  6. G + H merge before Phase 6 (frontend rebuild).

Conflict avoidance

The known conflict: F + G both touch atlas.primitives.returns. Centralize that module FIRST (Finding 2.C from eng review), in its own micro-chunk before either lane needs it.

⚠ The two write traps

Trap 1: writing domain terms to decisions.jsonl instead of CONTEXT.md. decisions.jsonl is a chronological edit log; nobody reads it backward to find terminology. Terms go in CONTEXT.md.

Trap 2: writing chunk progress to CONTEXT.md. CONTEXT.md is a glossary, NOT a status board. Status goes in docs/v6/phases/{phase}/STATUS.md.
